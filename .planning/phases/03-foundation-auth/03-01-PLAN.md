    ```elixir
    defmodule Astraplex.Accounts do
      use Ash.Domain, otp_app: :astraplex

      resources do
        resource Astraplex.Accounts.User
        resource Astraplex.Accounts.Token
      end
    end
    ```

    **Token resource (lib/astraplex/accounts/token.ex):**
    Standard AshAuthentication.TokenResource with AshPostgres data_layer, table "tokens", repo Astraplex.Repo. Add policy bypass for AshAuthenticationInteraction. Follow Pattern 2 from research.

    **User resource (lib/astraplex/accounts/user.ex):**
    Follow Pattern 1 from research with these specifics:
    - `use Ash.Resource` with `data_layer: AshPostgres.DataLayer`, `extensions: [AshAuthentication]`, `authorizers: [Ash.Policy.Authorizer]`, `domain: Astraplex.Accounts`
    - Attributes: `uuid_primary_key :id`, `email` as `:ci_string` (allow_nil?: false, public?: true), `hashed_password` as `:string` (allow_nil?: false, sensitive?: true), `role` as `:atom` (constraints: [one_of: [:admin, :staff]], default: :staff, allow_nil?: false, public?: true), `status` as `:atom` (constraints: [one_of: [:active, :deactivated]], default: :active, allow_nil?: false, public?: true), `create_timestamp :inserted_at`, `update_timestamp :updated_at`
    - Authentication block: tokens enabled with `token_resource Astraplex.Accounts.Token`, signing_secret from Application.fetch_env(:astraplex, :token_signing_secret). Password strategy with `identity_field :email`, `hashed_password_field :hashed_password`, `confirmation_required? false`, `registration_enabled? false` (per locked decision: no self-signup)
    - Identity: `identity :unique_email, [:email]`
    - Custom actions:
      - `create :create_user` — accepts email, password, password_confirmation, role. This is the admin-only user creation action. Accept `:role` argument. Set `allow_nil? false` for email/password.
      - `update :update_role` — accepts `:role` argument only
      - `update :deactivate` — `change set_attribute(:status, :deactivated)`. Add a change module that revokes all tokens for the user (query TokenResource for user's tokens and revoke them, or use AshAuthentication token revocation API if available; if not straightforward, invalidate by updating the user's `updated_at` which changes the token subject)
      - `update :reactivate` — `change set_attribute(:status, :active)`
      - `read :read` — basic read for admin listing
      - `read :get_by_email` — `get_by [:email]`
    - Sign-in preparation: Add a validation or preparation on the auto-generated `:sign_in_with_password` action that checks `status == :active`. If status is :deactivated, return same generic error as invalid credentials (never reveal account existence). This can be done via `sign_in_tokens_enabled? true` and adding a change/preparation.
    - Policies:
      - `bypass AshAuthentication.Checks.AshAuthenticationInteraction` — authorize_if always() (needed for auth actions)
      - `policy action(:create_user)` — authorize_if `expr(^actor(:role) == :admin)`
      - `policy action(:update_role)` — authorize_if `expr(^actor(:role) == :admin)`
      - `policy action(:deactivate)` — authorize_if `expr(^actor(:role) == :admin)`
      - `policy action(:reactivate)` — authorize_if `expr(^actor(:role) == :admin)`
      - `policy action(:read)` — authorize_if `expr(^actor(:role) == :admin)` (for now; will expand in later phases)
      - `policy action(:get_by_email)` — authorize_if `expr(^actor(:role) == :admin)`
    - Postgres: table "users", repo Astraplex.Repo

    **Run migration:** `mix ash.codegen create_accounts_domain` then `mix ash.migrate`.

    **Smokestack factory (test/support/factory.ex):**
    Add a User factory using `Ash.Seed.seed!` pattern (not Ash actions, since factory is for prerequisite data). The factory should create users with pre-hashed passwords using `AshAuthentication.BcryptProvider.hash/1` or `Bcrypt.hash_pwd_salt/1`.

    Add `register_and_log_in_user/2` function per research example. Creates a user with Ash.Seed.seed!, signs in via AshAuthentication strategy, stores in session on conn. Accept opts for role and status.

    - Run tests to confirm RED, then implement, then confirm GREEN.
