# Phase 3.1: UI Patterns - Research

**Researched:** 2026-03-10
**Domain:** Phoenix LiveView layout composition, daisyUI component patterns, responsive navigation
**Confidence:** HIGH

## Summary

Phase 3.1 replaces Phoenix boilerplate layouts with a real messaging app shell. The core challenge is role-based layout composition in Phoenix 1.8, which has a clean solution: define multiple layout function components in the Layouts module and call them explicitly in each LiveView's render function. DaisyUIComponents 0.9.3 provides most building blocks (sidebar, drawer, navbar, dropdown, avatar, collapse, menu, tabs, card, modal, indicator, loading), and daisyUI 5 provides the `dock` CSS component for mobile bottom navigation and `skeleton` for loading states.

The existing codebase has a solid foundation: LiveAuth on_mount hooks for role guarding, ash_authentication_live_session for auth scoping, DaisyUIComponents already imported globally via `use DaisyUIComponents`, and a corporate + dark theme already configured. The primary work is replacing the default `Layouts.app/1` with composable shell components and establishing the shared component module.

**Primary recommendation:** Define role-specific layout functions (e.g., `Layouts.staff_shell/1`, `Layouts.admin_shell/1`) in the existing Layouts module, each composing a shared base shell component. Each LiveView explicitly calls its layout in render/1. Use daisyUI's `dock` for mobile navigation and Tailwind responsive classes (`md:hidden`, `hidden md:flex`) for breakpoint switching.

### Pattern 1: Role-Based Layout Composition via Phoenix 1.8 Function Components

**What:** Define multiple layout functions in Layouts module. Each LiveView explicitly selects its layout by calling the appropriate function component in its render/1.

**When to use:** Always. This is the core architectural pattern for the entire app.

**Why this approach:**
- Phoenix 1.8 made `Layouts.app/1` an explicit function component call in render/1 (not magic)
- You can define `Layouts.admin_shell/1` and `Layouts.staff_shell/1` alongside `Layouts.app/1`
- Each LiveView chooses its layout explicitly -- no runtime conditional rendering
- on_mount hooks CANNOT set layouts (confirmed via Phoenix LiveView issue #2071, resolved by PR #2074 for live_session only)
- live_session CAN set layout, but that means separate live_sessions per role which prevents navigation between them
- Function component approach is most flexible and explicit

**Confidence:** HIGH -- verified against Phoenix 1.8 release notes, ElixirCasts tutorial, and LiveView docs.

**Example:**
```elixir
# lib/astraplex_web/components/layouts.ex

# Shared base shell -- top bar + flash messages + responsive container
defp base_shell(assigns) do
  ~H"""
  <.top_bar current_user={@current_user} />
  <div class="flex h-[calc(100vh-4rem)]">
    {render_slot(@inner_block)}
  </div>
  <.flash_group flash={@flash} />
  """
end

# Admin shell -- base + sidebar with admin links visible
attr :flash, :map, required: true
attr :current_user, :map, required: true
slot :inner_block, required: true

def admin_shell(assigns) do
  ~H"""
  <.base_shell flash={@flash} current_user={@current_user}>
    <.app_sidebar current_user={@current_user} role={:admin} class="hidden md:flex" />
    <main class="flex-1 overflow-y-auto">
      {render_slot(@inner_block)}
    </main>
    <.mobile_dock current_user={@current_user} role={:admin} class="md:hidden" />
  </.base_shell>
  """
end

# Staff shell -- base + sidebar without admin links
def staff_shell(assigns) do
  ~H"""
  <.base_shell flash={@flash} current_user={@current_user}>
    <.app_sidebar current_user={@current_user} role={:staff} class="hidden md:flex" />
    <main class="flex-1 overflow-y-auto">
      {render_slot(@inner_block)}
    </main>
    <.mobile_dock current_user={@current_user} role={:staff} class="md:hidden" />
  </.base_shell>
  """
end
```

```elixir
# In a LiveView's render/1
def render(assigns) do
  ~H"""
  <Layouts.admin_shell flash={@flash} current_user={@current_user}>
    <.page_header title="User Management">
      <:actions>
        <.link navigate={~p"/admin/users/new"} class="btn btn-primary btn-sm">New User</.link>
      </:actions>
    </.page_header>
    <%-- page content --%>
  </Layouts.admin_shell>
  """
end
```

**Important nuance:** The CONTEXT.md says "same shell layout for admin -- sidebar stays identical when admin pages are active." This means admin and staff shells share the same sidebar structure, but the admin shell shows the Admin nav link while staff shell hides it. This is NOT a separate admin layout -- it is the same shell with role-conditional sidebar items. The simplest implementation: one `app_shell/1` function that receives the `current_user` and conditionally shows the Admin link based on role. Two separate shell functions (admin_shell/staff_shell) are cleaner per the "not conditional rendering" constraint.

### Pattern 3: Collapsible Sidebar Groups

**What:** Channels, DMs, Groups sections are collapsible/expandable in the sidebar.

**When to use:** Sidebar messaging navigation sections.

**Implementation options:**
1. **daisyUI collapse component** -- available via DaisyUIComponents.Collapse. Uses checkbox toggle, CSS-only, no JS needed.
2. **HTML details/summary** -- native browser element, simplest, accessible by default.
3. **LiveView JS commands** -- JS.toggle for show/hide with animations.

**Recommendation:** Use HTML `<details>/<summary>` elements styled with Tailwind. They are accessible, require no JavaScript, persist open/closed state across re-renders naturally, and are the simplest solution. The daisyUI collapse component works but adds unnecessary markup. LiveView JS commands would require tracking open/closed state in assigns.

**Example:**
```elixir
<details open class="group">
  <summary class="flex items-center justify-between px-3 py-2 cursor-pointer text-sm font-semibold text-base-content/70 hover:text-base-content">
    Channels
    <.icon name="hero-chevron-down" class="size-4 transition-transform group-open:rotate-180" />
  </summary>
  <ul class="menu menu-sm pl-2">
    <%-- channel items --%>
  </ul>
</details>
```

### Pattern 4: Bottom Sheet for Mobile "Apps" Menu

**What:** The Apps button in the mobile dock opens a bottom sheet showing available pages.

**Implementation:** Use daisyUI modal with `modal-bottom` class. This slides up from the bottom on mobile and is the closest daisyUI equivalent to a bottom sheet. No custom CSS needed.

**Example:**
```elixir

  <ul class="menu">
    <li><.link navigate={~p"/admin/users"}>Admin</.link></li>
    <%-- more items --%>
  </ul>
</.modal>
```

### Anti-Patterns to Avoid
- **Conditional rendering in a single layout function:** Do NOT use `if @current_user.role == :admin` inside a shared layout. Per CONTEXT.md, role layouts compose independently from a base shell.
- **Separate live_sessions per role for layout purposes:** This prevents LiveView navigation between admin and non-admin routes (requires full page reload). Keep the existing router structure.
- **Using DaisyUIComponents.Sidebar directly:** The built-in sidebar component has its own toggle/collapse behavior and fixed widths that conflict with the app's requirements (fixed sidebar, no collapse on desktop, bottom dock on mobile). Build a custom sidebar using the primitives (menu, collapse, dropdown).
- **Storing sidebar collapse state in LiveView assigns:** Use CSS-only solutions (details/summary, Tailwind responsive) for UI-only state to avoid server roundtrips.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Bottom navigation | Custom fixed-bottom nav | daisyUI `dock` component | Handles spacing, iOS safe areas, active states |
| Dropdown menus | Custom popover with JS | DaisyUIComponents `dropdown` | Handles positioning, click-outside, accessibility |
| Collapsible sections | Custom JS toggle | HTML `<details>/<summary>` | Native, accessible, no JS, persists state |
| Modal dialogs | Custom overlay system | DaisyUIComponents `modal` | Handles backdrop, escape key, focus trap |
| Loading spinners | Custom CSS animation | DaisyUIComponents `loading` | Multiple shapes, consistent sizing |
| Skeleton loading | Custom skeleton CSS | daisyUI `skeleton` class | animate-pulse built in, consistent with theme |
| Avatar with indicator | Custom positioned dot | DaisyUIComponents `avatar` + `indicator` | Online/offline states, proper positioning |
| Tabs | Custom tab switching | DaisyUIComponents `tabs` | Accessible tab roles, consistent styling |
| Theme toggle | New implementation | Existing theme_toggle (relocate from Layouts to dropdown) | Already works with localStorage, JS dispatch |
| Badge/indicator | Custom badge | DaisyUIComponents `badge` + `indicator` | Colors, positioning on dock/sidebar items |

**Key insight:** DaisyUIComponents 0.9.3 covers nearly every primitive needed. The work is composing these into app-level patterns, not building new primitives.

## Common Pitfalls

### Pitfall 1: LiveView Layout Confusion with Phoenix 1.8
**What goes wrong:** Developers try to set layouts via on_mount hooks or expect the old `use Phoenix.LiveView, layout: ...` pattern to work.
**Why it happens:** Phoenix 1.8 changed how layouts work. The app layout is now an explicit function component call.
**How to avoid:** Always call the layout function component explicitly in render/1. Do not try to set it via on_mount or use directives.
**Warning signs:** Layout not rendering, blank pages, or double-rendering of layout elements.

### Pitfall 2: Mobile Dock and Content Overlap
**What goes wrong:** Fixed bottom dock overlaps page content, cutting off the last items in scrollable lists.
**Why it happens:** The dock is position-fixed but content doesn't account for its height.
**How to avoid:** Add `pb-16 md:pb-0` (or appropriate padding) to the main content area on mobile to create space below the dock.
**Warning signs:** Last list items hidden behind dock, especially on iOS.

### Pitfall 3: iOS Safe Area Insets
**What goes wrong:** Dock buttons are partially hidden behind the iOS home indicator.
**Why it happens:** iOS has safe area insets at the bottom that need explicit handling.
**How to avoid:** Add `<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">` to root.html.heex. The daisyUI dock documentation explicitly requires this. Use `env(safe-area-inset-bottom)` padding.
**Warning signs:** Dock looks fine on desktop browser but clipped on mobile Safari.

### Pitfall 4: LiveView Navigation Between Shells
**What goes wrong:** Navigating between admin and staff pages causes full page reload instead of smooth LiveView navigation.
**Why it happens:** Using separate live_sessions for admin and staff routes causes this. LiveView can only navigate within the same live_session without a full reload.
**How to avoid:** Keep the existing router structure. Admin routes and staff routes should share the same ash_authentication_live_session where possible. The on_mount hooks handle authorization. Layout selection happens in render/1, not at the router level.
**Warning signs:** Page flashes white during admin navigation, loss of WebSocket connection on route change.

### Pitfall 5: Sidebar Width and Content Overflow
**What goes wrong:** Long channel names or user names overflow the sidebar, causing horizontal scroll or layout breaks.
**Why it happens:** Fixed-width sidebar with variable-length content.
**How to avoid:** Use `truncate` class on text elements, set `overflow-hidden` on the sidebar container. Use `title` attribute on truncated items for hover tooltip.
**Warning signs:** Horizontal scrollbar appearing on the page.

### Pitfall 6: Theme Toggle Relocation
**What goes wrong:** Moving the theme toggle from the main layout to the user dropdown breaks the JS event dispatch.
**Why it happens:** The theme toggle uses `phx-click={JS.dispatch("phx:set-theme")}` which dispatches a DOM event. The existing JS listener in root.html.heex handles it. This should still work regardless of where the toggle lives in the DOM.
**How to avoid:** Keep the same JS.dispatch pattern. Just move the component markup into the dropdown. Verify the window-level event listener still catches it.
**Warning signs:** Theme not switching when clicking toggle in dropdown.

### Shared Components Module (AstraplexWeb.Components.UI)

```elixir
# lib/astraplex_web/components/ui.ex
defmodule AstraplexWeb.Components.UI do
  @moduledoc "App-level reusable UI components for Astraplex."

  use Phoenix.Component

  # Page Header
  attr :title, :string, required: true
  slot :actions

  def page_header(assigns) do
    ~H"""
    <div class="flex items-center justify-between px-6 py-4 border-b border-base-300">
      <h1 class="text-xl font-bold">{@title}</h1>
      <div :if={@actions != []} class="flex items-center gap-2">
        {render_slot(@actions)}
      </div>
    </div>
    """
  end

  # Empty State
  attr :icon, :string, default: nil
  attr :title, :string, required: true
  attr :description, :string, default: nil
  slot :actions

  def empty_state(assigns) do
    ~H"""
    <div class="flex flex-col items-center justify-center py-12 text-center">
      <.icon :if={@icon} name={@icon} class="size-12 text-base-content/30 mb-4" />
      <h3 class="text-lg font-semibold text-base-content/70">{@title}</h3>
      <p :if={@description} class="mt-1 text-sm text-base-content/50 max-w-sm">{@description}</p>
      <div :if={@actions != []} class="mt-4">
        {render_slot(@actions)}
      </div>
    </div>
    """
  end

  # Avatar with Initials
  attr :user, :map, required: true
  attr :size, :string, default: "md", values: ~w(xs sm md lg)
  attr :show_indicator, :boolean, default: false
  attr :online, :boolean, default: false

  def user_avatar(assigns) do
    ~H"""
    <div class={[
      "avatar placeholder",
      @show_indicator && (@online && "avatar-online" || "avatar-offline")
    ]}>
      <div class={[
        "rounded-full text-white",
        avatar_bg_color(@user.id),
        avatar_size(@size)
      ]}>
        <span class={avatar_text_size(@size)}>
          {initials(@user.email)}
        </span>
      </div>
    </div>
    """
  end

  # Skeleton variants
  def skeleton_list(assigns) do
    ~H"""
    <div class="flex flex-col gap-3 p-4">
      <div :for={_i <- 1..5} class="flex items-center gap-3">
        <div class="skeleton h-10 w-10 shrink-0 rounded-full"></div>
        <div class="flex flex-col gap-2 flex-1">
          <div class="skeleton h-4 w-1/3"></div>
          <div class="skeleton h-3 w-2/3"></div>
        </div>
      </div>
    </div>
    """
  end

  def skeleton_card(assigns) do
    ~H"""
    <div class="flex flex-col gap-4 p-4">
      <div class="skeleton h-32 w-full rounded-lg"></div>
      <div class="skeleton h-4 w-3/4"></div>
      <div class="skeleton h-4 w-1/2"></div>
    </div>
    """
  end

  # Private helpers
  defp initials(email) do
    email
    |> to_string()
    |> String.split("@")
    |> hd()
    |> String.slice(0, 2)
    |> String.upcase()
  end

  defp avatar_bg_color(id) do
    colors = ~w(bg-primary bg-secondary bg-accent bg-info bg-success bg-warning bg-error)
    index = :erlang.phash2(id, length(colors))
    Enum.at(colors, index)
  end

  defp avatar_size("xs"), do: "w-6 h-6"
  defp avatar_size("sm"), do: "w-8 h-8"
  defp avatar_size("md"), do: "w-10 h-10"
  defp avatar_size("lg"), do: "w-14 h-14"

  defp avatar_text_size("xs"), do: "text-xs"
  defp avatar_text_size("sm"), do: "text-xs"
  defp avatar_text_size("md"), do: "text-sm"
  defp avatar_text_size("lg"), do: "text-lg"
end
```

### Sidebar Width Recommendation

**Recommendation: w-64 (16rem / 256px)**

This is the standard sidebar width used by Slack, Discord, and similar messaging apps. It provides enough space for channel names with badges while not consuming too much screen real estate. The DaisyUIComponents sidebar default (w-72) is slightly wider; w-64 is more appropriate for a messaging app.

### Inline Actions vs Click-to-Detail (Research Question #2)

**Recommendation: Use inline actions for status changes, click-to-detail for complex operations.**

Based on UX research:
- **Inline actions** (buttons visible in row/card): Best for simple, single-step operations like toggling role, deactivating a user, or archiving a channel. Already established in UserListLive. Keep this pattern.
- **Click-to-detail**: Best when the action requires additional context, multi-field forms, or reviewing related data. For a messaging app, clicking a conversation row should navigate to the conversation (click-to-detail). Channel settings or user profile editing should use drawers (side panels).
- **Row hover actions**: NOT recommended for this app. They are undiscoverable on touch devices and conflict with mobile-first design.

**Pattern for list views:**
- **Table lists (admin)**: Always-visible inline action buttons (existing pattern from UserListLive). Keep it.
- **Card/row lists (conversations)**: Click entire row to navigate. No inline actions on conversation rows -- the action IS navigation.

## Open Questions

1. **Router restructuring for shared live_session**
   - What we know: Currently admin routes use a separate `ash_authentication_live_session :admin` block. This means navigating between `/` (in `:authenticated` session) and `/admin/users` (in `:admin` session) causes a full page reload.
   - What's unclear: Whether merging into a single live_session is safe given the different on_mount hooks.
   - Recommendation: Keep separate live_sessions for now. Admin navigation is infrequent enough that a page reload is acceptable. Merging sessions is a future optimization. The on_mount `:require_admin` hook on admin routes still works within a shared session -- this CAN be merged but is out of scope for this phase.

2. **Sidebar state persistence across pages**
   - What we know: Collapse state for Channels/DMs/Groups sections should persist as users navigate.
   - What's unclear: Whether `<details open>` persists across LiveView patch navigation within the same live_session.
   - Recommendation: Start with `<details open>` (all sections expanded by default). If state doesn't persist on navigation, use `phx-hook` with localStorage as a progressive enhancement in a later phase.
