# Phase 03.1: UI Patterns - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Establish the app shell layout, sidebar navigation, responsive mobile dock, role-based composable layouts, and reusable component patterns that all feature phases (4-10) build on. Replaces the default Phoenix boilerplate layout with a real messaging app chrome. No messaging functionality — this is pure UI scaffolding.

</domain>

<decisions>
## Implementation Decisions

### App Shell Layout
- 2-panel layout: fixed-width sidebar + main content area
- Full-width top bar spanning entire viewport: "Astraplex" branding left, search input right
- Search input is a placeholder (search is v2 — SRCH-01/02), establishes visual pattern
- Top bar is a continuous horizontal bar, sidebar and content sit below it
- Sidebar is fixed width, not resizable
- Theme toggle (light/dark/system) lives inside a user avatar dropdown menu, not in the main layout

### Sidebar Navigation
- Top section: page links — Home and Admin (admin-only, guarded by role)
- Divider separating page links from messaging sections
- Collapsible groups below: Channels, Direct Messages, Groups — each section expandable/collapsible
- Presence dots on DM users (placeholder — Phase 7 wires real data)
- Unread count badges on conversations (placeholder — Phase 7 wires real data)
- Bottom of sidebar: user info area with avatar + name + dropdown (Sign Out, Theme Toggle)
- Route-based navigation: each conversation has its own URL (/channels/:id, /dm/:id, /groups/:id)

### Admin Chrome
- Same shell layout — sidebar stays identical when admin pages are active
- Admin link highlights as active in sidebar
- Main content area swaps to show admin pages
- Admin sub-pages (Phase 9: Users, Channels, Audit Log) use tabs within the content area
- No separate admin layout

### Responsive Behavior
- Breakpoint: Tailwind md (768px) — below is mobile, above is desktop
- Mobile: no sidebar — replaced by bottom dock navigation
- Bottom dock order: Home, Inbox, Apps, Create (as icons)
- "Apps" button opens a bottom sheet with available pages (Admin, Settings, etc.) — waffle menu
- Inbox tab shows full-screen conversation list (channels, DMs, groups mixed). Tap to open conversation full-screen, back arrow to return
- Unread badge (total count) on Inbox icon in dock
- Desktop: standard sidebar + content layout

### Role-Based Composable Layouts
- Base shell component: top bar, responsive container, flash messages
- Role-specific layout modules that compose the base shell (not conditional rendering)
- Admin layout: base shell + sidebar with admin nav links visible
- Staff layout: base shell + sidebar with staff-appropriate nav only
- Each role guarded by auth — separate layout composition per role
- Pattern must be extensible for future roles (Manager, Maintenance, etc. in v2)
- Research needed: best pattern for role-based layout composition in Phoenix LiveView

### Shared Components
- Single shared components module (AstraplexWeb.Components.UI) for app-specific patterns
- DaisyUIComponents handles primitives; this module handles app-level patterns
- Components to establish:
  - **Page header**: title + optional action buttons, no breadcrumbs
  - **Empty state**: standard component with icon, title, description, and action slot
  - **Loading skeletons**: function components (<.skeleton_list />, <.skeleton_card />, etc.)
  - **Avatar**: initials on deterministic colored background (based on user ID), online indicator dot, size variants

### Page-Level View Patterns
- Four page types to establish as reusable patterns:
  1. **List view (table)**: table with headers, rows, actions — extends existing UserListLive pattern
  2. **List view (card/row)**: card or row-based list for conversations, message previews
  3. **Detail/content view**: header + scrollable content, no right panel — metadata via button/popover if needed
  4. **Form view**: drawers for edit forms, modals for destructive action confirmations
- Research needed: best pattern for inline actions vs click-to-detail in list views
- Standard chat layout: messages scrolling up, input bar pinned at bottom, channel name in header
- Chat input: plain text only — no rich text editor (markdown support deferred to v2, replaces MSG-02 rich text)

### Claude's Discretion
- Exact sidebar width (px/rem)
- Dock icon choices and styling
- Bottom sheet animation and backdrop
- Skeleton component shape variants
- Color generation algorithm for avatar initials
- Exact spacing, typography, and padding within components
- Loading state transitions and animations

</decisions>

<specifics>
## Specific Ideas

- "Less is more" — user prefers simplicity over feature density
- Dock-based mobile navigation reduces actions needed to accomplish tasks — fewer taps to get anywhere
- Role layouts should NOT share logic with dynamic rendering — each role composes independently from a base shell
- Drawers for edit forms give more space than modals; modals reserved for destructive confirmations (deactivate, delete)
- Plain text messaging to start; markdown replaces rich text (TipTap) for v2 — removes TipTap + LiveView integration risk noted in STATE.md

</specifics>
