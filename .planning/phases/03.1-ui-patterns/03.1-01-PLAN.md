Create `lib/astraplex_web/components/ui.ex` as `AstraplexWeb.Components.UI` with `use Phoenix.Component`. Include these function components:

1. **page_header/1** - attr :title (string, required), slot :actions. Renders flex row with h1 title left, actions slot right, border-b border-base-300, px-6 py-4.

2. **empty_state/1** - attr :icon (string, optional), attr :title (string, required), attr :description (string, optional), slot :actions. Renders centered column with icon, title, description, action slot. Use Heroicons via `<.icon>` (available from DaisyUIComponents).

3. **user_avatar/1** - attr :user (map, required), attr :size (string, default "md", values ~w(xs sm md lg)), attr :show_indicator (boolean, default false), attr :online (boolean, default false). Renders daisyUI avatar placeholder with initials from email (first 2 chars before @, uppercased), deterministic background color from :erlang.phash2(user.id) over daisyUI color classes. Size variants: xs=w-6 h-6, sm=w-8 h-8, md=w-10 h-10, lg=w-14 h-14. For indicator, use daisyUI avatar online/offline classes.

4. **skeleton_list/1** - Renders 5 rows of circle + two lines skeleton with daisyUI `skeleton` class.

5. **skeleton_card/1** - Renders rectangle + two lines skeleton with daisyUI `skeleton` class.

Private helpers: `initials/1` (email to 2-char uppercase), `avatar_bg_color/1` (phash2 over color list), `avatar_size/1`, `avatar_text_size/1`.

Reference the RESEARCH.md code examples section for exact implementation details.

Then add `import AstraplexWeb.Components.UI` to the `html_helpers/0` function in `lib/astraplex_web.ex`, after the DaisyUIComponents use statement. This makes all UI components available in every template.

  <name>Task 2: Build role-based shell layouts with top bar, sidebar, and user dropdown</name>
  <files>lib/astraplex_web/components/layouts.ex, lib/astraplex_web/components/layouts/root.html.heex, test/astraplex_web/live/layout_test.exs</files>
  <action>
**root.html.heex:** Update viewport meta tag to `content="width=device-width, initial-scale=1, viewport-fit=cover"` for iOS safe area support (required by daisyUI dock). Remove the suffix from live_title (change `suffix=" . Phoenix Framework"` to no suffix or just `" . Astraplex"`). Keep the theme script as-is.

**layouts.ex:** Replace the entire module content. Keep `use AstraplexWeb, :html` and `embed_templates "layouts/*"`. Remove the old `app/1` function (no longer needed -- LiveViews will call shell functions directly). Relocate `theme_toggle/1` to remain in this module (it is used inside the user dropdown).

Build these private and public components:

**Private: `base_shell/1`** - Shared wrapper: top_bar + flex container for sidebar+content + flash_group. Takes flash, current_user, inner_block slot.

**Private: `top_bar/1`** - Full-width bar spanning viewport. Uses DaisyUIComponents navbar. Left side: `<.link navigate={~p"/"}>Astraplex</.link>` as text-xl font-bold. Right side: disabled search input (placeholder only, hidden on mobile via `hidden md:block`, input-sm input-bordered w-48). Height h-16, border-b border-base-300, shrink-0.

**Private: `app_sidebar/1`** - attr :current_user (map), attr :role (:admin | :staff), attr :class (string, default ""). Fixed width w-64, bg-base-200, border-r border-base-300, flex flex-col, overflow-y-auto. Structure:
  - Top: `<ul class="menu">` with Home link (`~p"/"`, hero-home icon) and Admin link (`~p"/admin/users"`, hero-shield-check icon, only rendered when role == :admin). Highlight active link based on current path using `@current_user` is NOT sufficient -- sidebar needs to know the current path. Pass it as an attr or use `assigns[:active_page]`.
  - Divider: `<div class="divider my-1"></div>`
  - Collapsible groups using HTML `<details open>` with `<summary>`: Channels, Direct Messages, Groups. Each with chevron-down icon that rotates on open via `group-open:rotate-180`. Content area shows `<ul class="menu menu-sm pl-2">` with placeholder text "(No channels yet)" etc. Each group has presence dots (placeholder spans) and unread badges (placeholder).
  - Bottom: User info area. Fixed at bottom via mt-auto. Shows user_avatar (from UI module) + user email (truncated) + dropdown with Sign Out link (`~p"/sign-out"`) and theme_toggle. Use DaisyUIComponents dropdown component with `dropdown-top dropdown-end` for upward opening.

**Public: `admin_shell/1`** - attrs: flash (map, required), current_user (map, required), active_page (atom, default nil), slot inner_block. Renders base_shell wrapping: app_sidebar(role: :admin, class: "hidden md:flex") + `<main class="flex-1 overflow-y-auto pb-16 md:pb-0">` with inner_block.

**Public: `staff_shell/1`** - Same as admin_shell but role: :staff on the sidebar.

**Keep `theme_toggle/1` as-is** (public function) -- it will be rendered inside the user dropdown in the sidebar.

**Important per CONTEXT.md:** "Same shell layout for admin -- sidebar stays identical when admin pages are active. Admin link highlights as active in sidebar." The admin_shell and staff_shell differ ONLY in whether the Admin link is shown. The sidebar structure is otherwise identical.

**Active page highlighting:** Accept an `active_page` attr on the shells (:home, :admin, :inbox). Use this to apply `menu-active` class to the corresponding sidebar link. LiveViews will pass this value.
